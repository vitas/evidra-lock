# Evidra — Code & Policy Review (hardcode + design integrity)
**Repo snapshot:** evidra-v0.1.zip (local extract)
**Date:** 2026-02-26

This report lists issues and improvement ideas found during a review of Go sources and the `ops-v0.1` OPA policy bundle.

---
## 1. Policy input expects input.actions, but ToolInvocation params validation forbids action/actions keys
**Severity:** CRITICAL

### Why this matters
- OPA policy rules iterate over input.actions in most deny/warn rules (e.g. ops-v0.1 bundle rules).
- Go policy engine builds input.actions only from inv.Params["action"] or inv.Params["actions"].
- ToolInvocation.ValidateStructure() rejects unknown params keys and only allows: target, payload, risk_tags, scenario_id.
- Result: in normal requests, input.actions will be missing → most rules won't trigger (silent allow).

### Evidence
- `pkg/policy/policy.go` — buildActionList reads params["action"] and params["actions"]
- `pkg/invocation/invocation.go` — allowedParamKeys excludes action/actions
- `policy/bundles/ops-v0.1/evidra/policy/rules/*.rego` — rules use input.actions

### Recommended fix
- Pick ONE canonical contract and align all layers:
- Option A (recommended): represent actions inside params.payload (already canonical), and change buildActionList to also accept payload.action(s) or payload.actions; OR build actions from payload directly.
- Option B: extend allowedParamKeys to include "action" and/or "actions" and update docs accordingly (less clean, but minimal).
- Add an integration test: send a minimal invocation with payload-only and verify at least one deny rule fires when expected.

## 2. Static tenant id hardcoded for Phase 0 auth
**Severity:** MEDIUM

### Why this matters
- StaticTenantID is hardcoded to "static" in auth middleware.
- Not a secret, but it bakes a multi-tenant placeholder into runtime behavior and will leak into evidence/audit unless replaced later.

### Evidence
- `internal/auth/middleware.go:15` — `	staticTenantID = "static"`

### Recommended fix
- Keep for Phase 0 if you want, but isolate it behind config/env (EVIDRA_TENANT_ID_DEFAULT) or attach a TODO with a tracked issue.
- Ensure evidence/policy decisions don't rely on tenant_id semantics before multi-tenancy lands.

## 3. Hardwired embedded policy bundle (ops-v0.1) for zero-config MCP
**Severity:** MEDIUM

### Why this matters
- bundleembed.go embeds policy/bundles/ops-v0.1 at compile time and positions it as default for evidra-mcp.
- This is convenient, but it hard-couples MCP runtime to a single bundle/version and can cause drift between deployed API bundle and MCP bundle.

### Evidence
- `bundleembed.go:1` — `// Generated by AI | 2026-02-25 | Zero-config embedded OPA bundle for evidra-mcp | zero-config-embedded-bundle`
- `bundleembed.go:8` — `//go:embed all:policy/bundles/ops-v0.1`

### Recommended fix
- Make embedded bundle optional (fallback) and prefer explicit bundle selection via config/flag.
- Expose bundle revision/hash in responses (Decision.BundleRevision) and log it so drift is visible.
- If keeping embedded bundle: document compatibility guarantees and update process.

## 4. Policy rules depend on hardcoded action.kind strings (terraform.plan, k8s.apply, argocd.sync, etc.)
**Severity:** LOW

### Why this matters
- Rules match on action.kind == "terraform.plan" / "k8s.apply" etc.
- This is fine if action.kind is a stable contract, but it is a form of coupling/hardcoding that must be documented and versioned.

### Evidence
- `policy/bundles/ops-v0.1/evidra/policy/rules/deny_sg_open_world.rego:8` — `action.kind == "terraform.plan"`
- `policy/bundles/ops-v0.1/evidra/policy/rules/deny_dangerous_capabilities.rego:8` — `action.kind == "k8s.apply"`

### Recommended fix
- Document the canonical action.kind taxonomy in one place (docs/contracts_action_kind.md).
- Consider deriving kind from ToolInvocation.tool/operation in the server adapter layer to reduce client responsibility.
- Add a schema / enum validation for kind to prevent typos silently disabling rules.

## 5. Generated-by-AI headers in rego/go reduce trust and create noise
**Severity:** LOW

### Why this matters
- Many rego modules start with '# Generated by AI | <date> | ...'.
- Not functionally wrong, but it adds churn and may reduce reviewer confidence (also dates look like 'facts' inside policy).

### Evidence
- `policy/bundles/ops-v0.1/evidra/policy/decision.rego:1` — `# Generated by AI | 2026-02-24 | OPA bundle migration — decision aggregator | implementation_opa_bundles_all_phases.md`
- `pkg/invocation/invocation.go:12` — `	KeyRiskTags   = "risk_tags"`

### Recommended fix
- Replace with normal comments: purpose + ownership + link to design doc.
- If you want provenance: keep it in git history or in a single GENERATED.md rather than every file header.

## 6. Environment naming consistency: 'prod' vs 'production'
**Severity:** LOW

### Why this matters
- Params use by_env.production overrides for some keys.
- Tests and examples sometimes use environment: 'prod' (e.g., deny_prod_without_approval_test).
- If real invocations send 'prod', production overrides won't apply (defaults will).

### Evidence
- `policy/bundles/ops-v0.1/evidra/data/params/data.json` — production overrides present for ops.mass_delete.max_deletes, argocd.*, aws_s3.*
- `policy/bundles/ops-v0.1/tests/deny_prod_without_approval_test.rego:10` — `"environment": "prod",`

### Recommended fix
- Normalize environment names at ingress (CLI/server): map prod* → production.
- Or add by_env.prod mirrors in data.json for keys that have production overrides.
