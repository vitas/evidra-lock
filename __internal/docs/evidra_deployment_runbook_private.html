Traefik v3.3 Edge

Auto-discovery via Docker labels. Let's Encrypt ACME (TLS-ALPN-01 challenge). HSTS (1 year). Rate limiting middleware. Optional dashboard behind basic auth on traefik.${DOMAIN}.

evidra-api Core

Stateless Go binary. Embedded SPA (UI) + REST API on :8080. Ed25519 evidence signing with persistent key from EVIDRA_SIGNING_KEY. Feature-gated key issuance.

PostgreSQL 17 Data

Stores tenants, hashed API keys, skill definitions, executions, usage counters. Data on Hetzner Volume — survives VM rebuild.

Block Volume Storage

20 GB ext4 at /mnt/data. PostgreSQL + Traefik certs + evidence JSONL. Online resize: terraform apply with new volume_size.

3.2 Traefik Configuration

Traefik is configured entirely via Docker labels on the evidra-api service:

LabelPurpose

5.3 Dogfooding — Evidra Validates Evidra

When a PR to evidra-infra modifies terraform/**, the infra-validate workflow runs the Terraform plan through Evidra's own policy engine.

infra-validate.yml — Dogfooding Pipeline
sequenceDiagram
  participant PR as Pull Request
(evidra-infra)
  participant GH as GitHub Actions
  participant TF as Terraform
  participant AD as evidra-adapter-terraform
(pinned v0.1.0 + checksum)
  participant API as evidra.rest

  PR->>GH: PR touches terraform/**
  GH->>GH: curl adapter binary
verify SHA-256 checksum
  GH->>TF: terraform init + plan
  TF-->>GH: tfplan.json

  GH->>AD: stdin: tfplan.json
  AD-->>GH: {input: {...}, metadata: {...}}

  GH->>API: POST /v1/validate
Bearer EVIDRA_API_KEY
  API-->>GH: {allow, risk_level, reason}

  GH->>GH: Write to Step Summary
(plan counts + policy result)

  alt Policy allows
    GH-->>PR: ✅ Check passed
  else Policy denies
    GH-->>PR: ❌ Check failed
(reason in summary)
  end
Version pinning: Adapter version is set in env.ADAPTER_VERSION: v0.1.0. Checksum verified against checksums.txt from the same release. Update manually when upgrading.

6 Operations

6.1 Manual Deploy

# SSH to server
ssh root@<server-ip>
cd /opt/evidra

# Pull latest and restart
docker compose -f docker-compose.prod.yml pull
docker compose -f docker-compose.prod.yml up -d

# Check logs
docker compose -f docker-compose.prod.yml logs -f evidra-api
docker compose -f docker-compose.prod.yml logs -f traefik

# Verify health
curl -s https://evidra.rest/healthz | jq .

Server resize = recreation. The VM is destroyed and recreated with the new type. Data on the Hetzner Volume survives. Docker containers need to be restarted after. Cloud-init or manual SSH to start compose again.

6.3 Backup & Recovery

# PostgreSQL backup (on server)
docker compose -f docker-compose.prod.yml exec postgres \
  pg_dump -U evidra evidra | gzip > /mnt/data/backups/evidra_$(date +%Y%m%d).sql.gz

# Evidence backup (just copy JSONL files)
cp -r /mnt/data/evidence /mnt/data/backups/evidence_$(date +%Y%m%d)

# Restore PostgreSQL
gunzip -c backup.sql.gz | docker compose exec -T postgres psql -U evidra evidra

Hetzner also offers automatic server snapshots (~€0.01/GB/mo) as an additional backup layer.

6.5 Teardown

Destructive operation. This deletes all infrastructure including data. Ensure backups are taken first.
# Stop containers first
ssh root@<server-ip> "cd /opt/evidra && docker compose down"

# Destroy infrastructure
cd terraform
terraform destroy
