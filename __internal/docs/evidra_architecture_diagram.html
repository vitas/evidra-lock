1 System Overview

Evidra is a policy-as-code guardrail engine for AI agents and CI pipelines. It evaluates whether a tool invocation (Terraform apply, Kubernetes deploy, database migration) should be allowed, denied, or flagged — then produces a cryptographically signed evidence record.

Traefik v3 Edge

TLS termination via Let's Encrypt ACME. HTTP → HTTPS redirect. HSTS, rate limiting, security headers via middleware labels. Optional dashboard behind basic auth.

evidra-api Core

Stateless Go binary. Serves UI (embedded SPA) + REST API on :8080. Ed25519 evidence signing. Feature-gated key issuance via invite secret.

PostgreSQL 17 Data

Stores tenants, hashed API keys, skill definitions, execution log, usage counters. Data on Hetzner Block Volume — survives VM rebuild.

Block Volume Storage

20 GB ext4 at /mnt/data. Contains postgres data, Traefik certs, evidence JSONL. Online resize available.

3 Interaction Flows

3.1 AI Agent → MCP → Validate

When Claude Code (or any MCP-compatible agent) wants to run a tool like terraform apply, it first asks Evidra for permission via the MCP protocol.

MCP Validation Flow — AI Agent Guardrail
sequenceDiagram
  participant Agent as AI Agent
(Claude Code)
  participant MCP as evidra-mcp
(stdio)
  participant V as pkg/validate
  participant OPA as OPA Engine
  participant EV as Evidence Store

  Agent->>MCP: tools/call "validate"
{tool, operation, params}
  MCP->>V: EvaluateScenario()
  V->>OPA: Evaluate policy
  OPA-->>V: Decision{allow, risk_level, reasons}
  V->>EV: Append evidence record (signed)
  EV-->>V: event_id
  V-->>MCP: {allow: true/false, evidence_id}
  MCP-->>Agent: MCP result

  alt Allowed
    Agent->>Agent: Execute tool
  else Denied
    Agent->>Agent: Abort + explain to user
  end
Modes: enforce (default) — deny blocks execution. observe — policy evaluated and recorded, but never blocks. Useful for rollout.

3.2 CI Pipeline → Adapter → API

CI pipelines use the evidra-adapter-terraform binary to transform raw plan JSON into structured parameters, then POST to the API.

CI Pipeline Flow — Terraform Plan Validation
sequenceDiagram
  participant GH as GitHub Actions
  participant TF as terraform
  participant AD as evidra-adapter-terraform
  participant API as evidra-api
(evidra.rest)
  participant OPA as Policy Engine

  GH->>TF: terraform plan -out=tfplan.bin
  TF-->>GH: tfplan.bin
  GH->>TF: terraform show -json tfplan.bin
  TF-->>GH: plan.json (raw)

  GH->>AD: stdin: plan.json
  AD->>AD: Parse, extract counts,
filter, truncate
  AD-->>GH: stdout: {input, metadata}

  GH->>API: POST /v1/validate
Bearer token + input payload
  API->>OPA: Evaluate policy rules
  OPA-->>API: Decision
  API-->>GH: {allow, risk_level, reason}

  alt Allowed
    GH->>GH: ✅ Continue to terraform apply
  else Denied
    GH->>GH: ❌ Fail CI step
  end
Purity boundary: The adapter binary has NO network access. It only transforms data (stdin → stdout). The CI workflow is responsible for the API call.

Tenant model: Identity derived entirely from API key. Each key belongs to one tenant. Keys are hashed (SHA-256) at rest — the server never stores plaintext keys.

Adapter Binary Pure

evidra-adapter-terraform — transform only. No network. Testable offline. Cacheable (same input → same output). Exit codes: 0 success, 1 parse error, 2 usage error.

Future: evidra CLI Planned

Composes adapter + API call in one command: evidra ci terraform --plan tfplan.bin. Imports adapter as Go library, handles auth, prints summary. Ships after API stabilizes.

TierFieldsFilter BehaviorTruncation
1 — Counts create_count, update_count, destroy_count, replace_count, total_changes Scope-filtered only. Never affected by filter_actions Never
2 — Risk delete_types, replace_types, delete_addresses, replace_addresses Scope-filtered. Independently truncated Own _total + _truncated
3 — Detail resource_changes[] (address, type, action, provider) Scope + action filtered max_resource_changes
Critical semantic rule: filter_resource_types narrows scope (affects counts). filter_actions narrows detail only (affects ONLY resource_changes array, never counts). Policy rules should prefer counts over iterating arrays.

The policy bundle is embedded in the binary at build time (zero-config mode). When --bundle is not specified, the server extracts the built-in ops-v0.1 bundle to a temp directory at startup.

Decision output contract: {allow, risk_level ("low"|"medium"|"high"), reason, reasons, hits, hints}. Risk is "high" when any deny rule fires, "medium" for breakglass-tagged allowed flows, "low" otherwise.

7.3 Dogfooding — Evidra Validates Evidra

When a PR to evidra-infra modifies Terraform files, CI runs the plan through Evidra's own policy engine. This is the "Evidra validates Evidra" story.

Dogfooding CI — infra-validate.yml
sequenceDiagram
  participant PR as Pull Request
  participant GH as GitHub Actions
  participant TF as Terraform
  participant AD as evidra-adapter-terraform
(pinned v0.1.0)
  participant API as evidra.rest

  PR->>GH: PR touches terraform/**
  GH->>GH: Download adapter binary
(pinned version + checksum verify)
  GH->>TF: terraform init && plan
  TF-->>GH: tfplan.json
  GH->>AD: stdin: tfplan.json
  AD-->>GH: {input: {destroy_count: 0, ...}}
  GH->>API: POST /v1/validate
  API-->>GH: {allow: true, risk_level: "low"}
  GH->>GH: ✅ Write to Step Summary

  Note over PR: PR shows: "Evidra policy check passed"
with plan summary (creates, destroys, etc.)
Pinned, not latest: The adapter version is pinned in the workflow (ADAPTER_VERSION: v0.1.0) with SHA-256 checksum verification. Reproducible builds — CI never breaks from an upstream adapter release.
