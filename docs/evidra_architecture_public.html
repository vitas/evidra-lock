1 System Overview

Evidra is a policy-as-code guardrail engine for AI agents and CI pipelines. It evaluates whether a tool invocation — Terraform apply, Kubernetes deploy, database migration, API call — should be allowed, denied, or flagged, then produces a cryptographically signed evidence record proving what was decided and why.

evidra/evidra Core

API server, MCP server, offline CLI, OPA policy engine, evidence store. Single Go module. Stateless API — all state lives in PostgreSQL and the evidence log.

evidra/adapters Separate Module

Adapter interface + tool-specific implementations (Terraform first). Pure functions: bytes in, JSON out. No network, no state. Independent release cycle.

Zero coupling: The server never imports adapter code. The adapter never imports server code. They communicate solely via the /v1/validate HTTP API with a shared JSON contract (skill input_schema).

2 Interaction Flows

2.1 AI Agent → MCP → Validate

When an MCP-compatible agent wants to execute a tool, it first asks Evidra for permission. The MCP server evaluates the policy and records a signed evidence entry before responding.

MCP Validation Flow — AI Agent Guardrail
sequenceDiagram
  participant Agent as AI Agent
(Claude Code)
  participant MCP as evidra-mcp
(stdio)
  participant V as pkg/validate
  participant OPA as OPA Engine
  participant EV as Evidence Store

  Agent->>MCP: tools/call "validate"
{tool, operation, params}
  MCP->>V: EvaluateScenario()
  V->>OPA: Evaluate policy
  OPA-->>V: Decision{allow, risk_level, reasons}
  V->>EV: Append evidence record (signed)
  EV-->>V: event_id
  V-->>MCP: {allow: true/false, evidence_id}
  MCP-->>Agent: MCP result

  alt Allowed
    Agent->>Agent: Execute tool
  else Denied
    Agent->>Agent: Abort + explain to user
  end
Modes: enforce (default) — deny blocks execution. observe (--observe flag) — policy is evaluated and recorded but never blocks. Useful for gradual rollout and shadow monitoring.

2.2 CI Pipeline → Adapter → API

CI pipelines use adapter binaries to transform raw tool output into structured parameters, then POST to the API for policy evaluation.

CI Pipeline Flow — Terraform Plan Validation
sequenceDiagram
  participant GH as CI Runner
  participant TF as terraform
  participant AD as evidra-adapter-terraform
  participant API as Evidra API
  participant OPA as Policy Engine

  GH->>TF: terraform plan -out=tfplan.bin
  TF-->>GH: tfplan.bin
  GH->>TF: terraform show -json tfplan.bin
  TF-->>GH: plan.json (raw)

  GH->>AD: stdin: plan.json
  AD->>AD: Parse, extract counts,
filter, truncate
  AD-->>GH: stdout: {input, metadata}

  GH->>API: POST /v1/validate
Bearer token + input payload
  API->>OPA: Evaluate policy rules
  OPA-->>API: Decision
  API-->>GH: {allow, risk_level, reason}

  alt Allowed
    GH->>GH: Continue to apply
  else Denied
    GH->>GH: Fail CI step
  end
Purity boundary: The adapter binary has NO network access. It only transforms data (stdin → stdout). The CI workflow is responsible for the API call. This means adapters work offline, in air-gapped environments, and in test pipelines.

2.3 Direct API Call

Platforms with structured data can POST directly to the API — no adapter needed. Two patterns are supported: raw validation and skill-based execution.

Direct API — Two Validation Patterns
sequenceDiagram
  participant P as Platform
(Backstage / Slack)
  participant V as POST /v1/validate
  participant SK as POST /v1/skills/{id}:execute

  Note over P,V: Pattern A: Raw validation
  P->>V: {tool, operation, params, context}
  V-->>P: {allow, risk_level, evidence_id}

  Note over P,SK: Pattern B: Skill execution
  P->>SK: {input: {destroy_count: 5, ...}}
  SK-->>P: {status, decision, evidence_id}

Pattern A is flexible — any tool, any operation. Pattern B uses pre-registered skill definitions with typed input schemas, input validation, and default parameters. Skills are registered via POST /v1/skills.

2.4 Evidence Signing & Verification

Every evaluation produces a signed evidence record. Records form a hash-linked chain (each record includes the hash of the previous), providing tamper-evident audit logs. Verification requires no authentication.

Evidence Lifecycle — Sign, Record, Verify
sequenceDiagram
  participant API as Evidra
  participant EV as Evidence Store
  participant AUD as Auditor

  Note over API: On every evaluation:
  API->>API: Create evidence record
(input, decision, timestamp)
  API->>API: Ed25519 sign record
  API->>EV: Append to JSONL
(hash-linked chain)

  Note over AUD: Verification (no auth required):
  AUD->>API: GET /v1/evidence/pubkey
  API-->>AUD: Ed25519 public key (cache this)
  AUD->>API: POST /v1/evidence/verify
{record}
  API-->>AUD: {valid: true/false}

  Note over AUD: Or verify fully offline:
pubkey + record → Ed25519.Verify()

Tenant model: Identity derived entirely from the API key. Each key belongs to one tenant. Keys are hashed (SHA-256) at rest — the server never stores plaintext keys. Key format: ev1_ + 32 crypto/rand bytes (base62), 48 chars total.

TierFieldsFilter BehaviorTruncation
1 — Counts create_count, update_count, destroy_count, replace_count, total_changes Scope-filtered only. Never affected by filter_actions Never
2 — Risk delete_types, replace_types, delete_addresses, replace_addresses Scope-filtered. Independently truncated per array Own _total + _truncated
3 — Detail resource_changes[] (address, type, action, provider) Scope + action filtered max_resource_changes (default 200)
Critical semantic rule: filter_resource_types narrows scope (affects counts, types, everything). filter_actions narrows detail only (affects ONLY resource_changes array, never counts). Policy rules should prefer counts over iterating arrays.

The policy bundle is embedded in the binary at build time (zero-config mode). When --bundle is not specified, the server extracts the built-in ops-v0.1 bundle to a temp directory at startup. Custom bundles can be loaded from any directory.

Decision output contract: {allow, risk_level, reason, reasons, hits, hints}. Risk is "high" when any deny rule fires, "medium" for breakglass-tagged allowed flows, "low" otherwise.

5.2 Rule Examples

# Block more than 10 deletions
deny[msg] {
    input.params.payload.destroy_count > 10
    msg := sprintf("Plan deletes %d resources — requires manual approval",
        [input.params.payload.destroy_count])
}

# Block deletion of volume types (uses risk shortcut — no iteration needed)
deny[msg] {
    "hcloud_volume" == input.params.payload.delete_types[_]
    msg := "Deleting volumes is blocked — data loss risk"
}

# Require manual review for truncated plans
deny[msg] {
    input.params.payload.resource_changes_truncated == true
    msg := sprintf("Plan has %d changes (truncated) — requires manual review",
        [input.params.payload.resource_changes_count])
}

Two deployment modes work without internet access:

MCP Server (fully offline)

The evidra-mcp binary with its embedded policy bundle runs entirely locally. No network needed. Evidence stored at ~/.evidra/evidence. Ideal for developer workstations in restricted environments.

API Server (internal network)

Deploy evidra-api from an internal container registry mirror. PostgreSQL on the same network. TLS via internal CA. CI pipelines connect to the internal endpoint.

6.6 Requirements & Sizing

ScaleEvals/dayCPURAMDisk (DB)API Replicas
Small< 1,0001 vCPU1 GB1 GB1
Medium1,000–50,0002 vCPU4 GB10 GB2
Large50,000+4+ vCPU8+ GB50+ GB3+

The API binary itself uses ~30 MB RAM at idle. OPA evaluation adds ~1 MB per concurrent evaluation. PostgreSQL dominates resource usage at scale. The evidence log grows at roughly 1 KB per evaluation.

Scaling strategy: The API server is stateless — add replicas behind a load balancer. PostgreSQL is the only stateful component. For high-scale deployments, use a managed PostgreSQL instance with read replicas.
