# ENGINE_V3_DOMAIN_ADAPTERS.md
## Unified Domain Adapter Architecture for Engine v3

## Status
Architecture proposal (merges ENGINE_V3_DOMAIN_ADAPTERS_ROADMAP +
DESIGN_GO_ADAPTER_CANONICALIZATION)

---

## 1. Why v3

Engine v2 achieved clean separation: normalized actions → domain-agnostic
rules → decision aggregation. Canonicalization lives in Rego and handles
one domain well: K8s workloads (podSpec-based resources).

v3 is needed because the four first-class tools each have distinct
canonicalization and intent identity needs:

- **kubectl** — K8s workload manifests (podSpec) work, but adding
  Service/Ingress/RBAC policies requires new non-podSpec
  canonicalization branches.

- **Helm** — needs release identity, chart identity, and values
  canonicalization for intent hashing. Values are the tricky part:
  the same chart with different `values.yaml` is a different intent.

- **Terraform** — plan metadata (v1 adapter) works for kill-switch
  rules. Deep extraction (v2 adapter for SG/IAM/S3) is the next step.
  External CLI adapter + MCP agent extraction are two paths.

- **ArgoCD** — app identity extraction beyond current minimal level.
  Sync parameters, target revision, and project context matter for
  intent identity and policy evaluation.

Without domain adapters, adding per-tool capabilities means domain
knowledge spreading across canonicalization, intent extraction, and
insufficient_context hinting — all tangled in shared code.

---

## 2. Architecture Decision: Hybrid (Rego + Go)

Neither pure-Rego nor pure-Go is correct. Each has strengths:

| Concern | Best in | Why |
|---|---|---|
| K8s workload canonicalization (kubectl) | Rego | Already works, ships in bundle, external OPA compatible |
| K8s non-workload canonicalization (Service, Ingress, RBAC) | Go | Different structures per kind, structured extraction with explicit field paths (no k8s.io/api) |
| Helm canonicalization | Go | Values hash, chart identity — not manifest parsing |
| Terraform canonicalization | Go (external) | Already in evidra-adapters repo, CLI pipeline |
| ArgoCD canonicalization | Go | App identity, sync params — simple extraction |
| Policy rules | Rego | Core competency, domain-agnostic flat evaluation |
| Intent key extraction | Go | Typed struct → hash, shared by deny cache |
| Missing data hints | Either | Rego for policy-derived hints, Go for structural errors |

### The hybrid model

```
Input arrives (raw manifest or plan)
    |
    v
Go adapter layer (pkg/adapter/)
    |
    +-- kubectl workload?                → pass through to Rego canonicalizer
    +-- kubectl non-workload (future)?   → Go extraction (Service, Ingress)
    +-- Helm?                            → Go extraction (release/chart/values)
    +-- Terraform?                       → external adapter or passthrough
    +-- ArgoCD?                          → Go extraction (app identity/sync)
    +-- Unknown?                         → pass through (fail-closed via policy)
    |
    v
OPA input: {actions: [flat maps]}
    |
    v
Rego rules evaluate (unchanged, domain-agnostic)
```

**Tool identity rule:** The `tool` field identifies the CLI that
executes the mutation. Manifest generators (kustomize, jsonnet, cdk8s,
cue) are not tools — they produce manifests that kubectl applies.
Agent should always send `tool=kubectl` for K8s operations regardless
of how the manifest was generated. If an agent sends an unknown tool
prefix, `deny_unknown_destructive` fires (fail-closed). The agent
contract and skill file must document this clearly.

**Agent confusion guardrail:** Agents will frequently send
`tool=kustomize` or `tool=helm` when applying K8s YAML that was
generated by those tools. The hosted contract (SKILL.md) and MCP
tool description MUST explicitly state: "If you used `helm template`
or `kustomize build` and then apply via kubectl, set `tool=kubectl`."
The engine itself does NOT alias or redirect tool names — if an agent
sends `tool=kustomize`, `deny_unknown_destructive` fires. This is
correct (fail-closed), but the UX depends on clear contract docs.

**Key insight:** K8s workload canonicalization stays in Rego. It works,
it's tested, it ships in bundles, and it's compatible with external OPA.
New domains get Go adapters because they need structured extraction
with explicit field paths — not Rego shape-detection heuristics. This
is unstructured Go extraction (manual field paths), NOT k8s.io/api
typed structs. The `k8s.io/api` dependency is explicitly excluded.

The Go adapter layer sits in `buildActionList()` — the function that
already transforms params into actions before OPA eval. It becomes a
dispatcher that checks the domain and routes accordingly.

## 3. Domain Adapter Interface

```go
// pkg/adapter/adapter.go

package adapter

// DomainAdapter handles canonicalization and intent extraction for a
// specific operational domain (k8s workloads, helm, terraform, etc).
//
// IMPORTANT: Adapters never read actor.* fields. Actor semantics
// (human/agent/ci gating) are handled by the decision layer.
type DomainAdapter interface {
    // Domain returns the adapter domain name (for logging/metrics).
    Domain() string

    // CanHandle returns true if this adapter should process the input.
    // Checked in priority order; first match wins.
    CanHandle(tool, operation string, payload map[string]any) bool

    // Canonicalize transforms raw input into flat action format.
    // Returns nil to signal "pass through to Rego canonicalizer."
    // Returns error for structural parse failures (precise messages).
    Canonicalize(tool, operation string, payload map[string]any) (*FlatAction, error)

    // IntentKey extracts semantic identity from the NORMALIZED action
    // for deny-loop prevention. Receives the flat payload AFTER
    // canonicalization (Rego or Go), not the raw input. This ensures
    // intent identity always matches what policy evaluates.
    //
    // Returns ("", nil) to signal "deny-loop not eligible" for this
    // action — cache is skipped, no blocking.
    IntentKey(tool, operation string, normalizedPayload map[string]any) (string, error)

    // MissingDataHints returns actionable hints for incomplete payloads.
    // Optional — return nil to use default insufficient_context logic.
    MissingDataHints(tool, operation string, payload map[string]any) []string
}

// FlatAction is the normalized format OPA rules expect.
type FlatAction struct {
    Tool        string         `json:"tool"`              // e.g. "kubectl"
    Operation   string         `json:"operation"`         // e.g. "apply"
    Kind        string         `json:"kind"`              // "tool.operation" for OPA compat
    Target      map[string]any `json:"target,omitempty"`
    RiskTags    []string       `json:"risk_tags,omitempty"`
    Payload     map[string]any `json:"payload"`
}

// ToMap converts to the map format OPA expects.
func (fa *FlatAction) ToMap() map[string]any {
    return map[string]any{
        "kind":      fa.Tool + "." + fa.Operation,
        "target":    fa.Target,
        "risk_tags": fa.RiskTags,
        "payload":   fa.Payload,
    }
}
```

**Note:** `Kind` is derived (`tool + "." + operation`) for OPA
compatibility. `Tool` and `Operation` are stored separately so
callers never have to parse a string back. Adapters set `Tool`
and `Operation` explicitly; `Kind` is computed by `ToMap()`.
```

**Critical design choice:** `IntentKey` receives the **normalized**
(flat) payload, not the raw input. This eliminates the risk of drift
between "what policy sees" and "what intent cache sees":

- For K8s workloads: Rego canonicalizes → Go reads flat output for
  intent key. No duplicate extraction logic.
- For Helm/ArgoCD/Terraform: Go canonicalizes → same Go code reads
  its own flat output for intent key. Trivially consistent.
- Native vs flat equivalence is guaranteed by the canonicalizer,
  not duplicated in the intent extractor.

### FlatAction.Payload per domain

All adapters produce the same flat field names that OPA rules already
read. This is the unifying contract:

```
kubectl (workloads):
                {namespace, resource, name, containers, init_containers,
                 volumes, host_pid, host_ipc, host_network}

kubectl (non-workload, future):
                Service:  {namespace, name, type, source_ranges}
                Ingress:  {namespace, name, tls_configured, rules}

Helm:           {namespace, release_name, chart, chart_version, values_hash}

Terraform:      {resource_types, destroy_count, security_group_rules,
                 iam_policy_statements, s3_public_access_block, ...}

ArgoCD:         {app_name, namespace, project, target_revision, sync_policy}
```

## 4. Domain Responsibilities

### 4.1 kubectl (K8s)

**Workload resources (Deployment, Pod, CronJob, StatefulSet, ...)**

Canonicalization: **Rego (stays as-is).**
The existing `canonicalize.rego` handles podSpec extraction from 5+
shapes. Go adapter returns `nil` → Rego takes over.

Intent extraction: **Go.**
```go
type K8sWorkloadAdapter struct{}

func (a *K8sWorkloadAdapter) CanHandle(tool, op string, payload map[string]any) bool {
    if tool != "kubectl" && tool != "oc" {
        return false
    }
    return hasWorkloadShape(payload)
}

func (a *K8sWorkloadAdapter) Canonicalize(_, _ string, _ map[string]any) (*FlatAction, error) {
    return nil, nil // Rego handles workload canonicalization
}

func (a *K8sWorkloadAdapter) IntentKey(tool, op string, normalizedPayload map[string]any) (string, error) {
    // Reads from FLAT payload (after Rego canonicalization).
    // Fields: namespace, resource, containers[].image,
    // host_pid, host_ipc, host_network, containers[].security_context
    intent := K8sWorkloadIntent{
        Namespace: getString(normalizedPayload, "namespace"),
        Resource:  getString(normalizedPayload, "resource"),
        Images:    extractSortedImages(normalizedPayload, "containers"),
        Posture:   extractPostureFromFlat(normalizedPayload),
    }
    return hashIntent(tool, op, intent), nil
}
```

**Non-workload resources (Service, Ingress, RBAC — future)**

When policies for these are added, they get separate adapters with
**Go canonicalization** (no podSpec to extract, different structures).
This is the point where Go adapters earn their keep for K8s.

### 4.2 Helm

**Canonicalization: Go.**

Helm payloads carry release identity, chart identity, and values.

```go
type HelmAdapter struct{}

func (a *HelmAdapter) CanHandle(tool, _ string, _ map[string]any) bool {
    return tool == "helm"
}

func (a *HelmAdapter) Canonicalize(tool, op string, payload map[string]any) (*FlatAction, error) {
    return &FlatAction{
        Tool:      tool,
        Operation: op,
        Payload: map[string]any{
            "namespace":    extractString(payload, "namespace"),
            "release_name": extractString(payload, "release_name", "release", "name"),
            "chart":        extractString(payload, "chart"),
            "chart_version": extractString(payload, "chart_version", "version"),
            "values_hash":  stableValuesHash(payload),
        },
    }, nil
}
```

**Intent key:** `release + namespace + chart + version + values_hash`.

Values hash tradeoff:
- Include → prevents retry loops, but blocks pure formatting changes
- Exclude → allows retry with silently modified values

**Decision:** Include stable values hash (sorted JSON of values map).
If you changed values, you should re-evaluate. Document this explicitly.

**Values hash guardrails:**

- **Size limit:** If serialized values exceed `maxValuesHashInputSize`,
  set `values_hash = "oversize"` and add hint: "Values too large for
  policy evaluation; provide rendered manifest via kubectl.apply
  instead." This prevents CPU/memory abuse on large Helm values.

  ```go
  // maxValuesHashInputSize caps the values payload size for hashing.
  // 64KB covers typical Helm values. Larger values indicate the agent
  // should send rendered manifests via kubectl.apply instead.
  // Configurable via adapter options if needed.
  const maxValuesHashInputSize = 64 * 1024
  ```

- **Secrets warning:** The values hash is computed from the values
  map the agent sends. Document in SKILL.md and MCP contract:
  "Do NOT include secrets or credentials in Helm values payload.
  Use secret references or external secret management."
  Evidra-Lock does not filter or redact values — it hashes what it
  receives. The contract must make this explicit.

- **YAML anchors/aliases:** The agent sends JSON (MCP transport).
  YAML anchors resolve before JSON serialization. Not a concern.

### 4.3 Terraform

**Canonicalization: External adapter (evidra-adapters repo) for CI.
Passthrough for MCP agent (agent provides flat payload directly).**

```go
type TerraformAdapter struct{}

func (a *TerraformAdapter) CanHandle(tool, _ string, _ map[string]any) bool {
    return tool == "terraform"
}

func (a *TerraformAdapter) Canonicalize(_, _ string, _ map[string]any) (*FlatAction, error) {
    return nil, nil // payload already flat (from agent or external adapter)
}

func (a *TerraformAdapter) IntentKey(tool, op string, normalizedPayload map[string]any) (string, error) {
    intent := TerraformIntent{
        ResourceTypes: extractSortedStrings(normalizedPayload, "resource_types"),
        DestroyCount:  extractIntOrNil(normalizedPayload, "destroy_count"),
        // v2 deep fields (when available):
        SGCIDRs:       extractSGCIDRs(normalizedPayload),
        IAMActions:    extractIAMActions(normalizedPayload),
    }
    return hashIntent(tool, op, intent), nil
}
```

**adapter-terraform v2** (deep extraction) ships independently in
the `evidra-adapters` repo. Produces `security_group_rules`,
`iam_policy_statements`, `s3_public_access_block` — fields the OPA
rules already know how to evaluate.

## 5. Adapter Registry and Dispatch

```go
// pkg/adapter/registry.go

type Registry struct {
    adapters []DomainAdapter // checked in order, first CanHandle wins
}

// AdapterResult captures which adapter handled the input.
// Returned by Canonicalize, reused by IntentKey — no second dispatch.
type AdapterResult struct {
    Adapter          DomainAdapter  // nil if no match
    Domain           string         // adapter domain name or "none"
    CanonicalizeMode string         // "rego" | "go" | "passthrough"
    IntentEligible   bool           // false if no adapter or empty key
}

func NewDefaultRegistry() *Registry {
    return &Registry{
        adapters: []DomainAdapter{
            &K8sWorkloadAdapter{},     // kubectl workloads
            // &K8sServiceAdapter{},   // future: Service, Ingress, RBAC
            &HelmAdapter{},
            &ArgocdAdapter{},
            &TerraformAdapter{},       // passthrough (agent/external adapter)
        },
    }
}

// Resolve finds the matching adapter for the RAW payload.
// Called once, before canonicalization. Returns AdapterResult that
// is reused for both Canonicalize and IntentKey — no second dispatch.
func (r *Registry) Resolve(tool, op string, rawPayload map[string]any) AdapterResult {
    for _, a := range r.adapters {
        if a.CanHandle(tool, op, rawPayload) {
            return AdapterResult{
                Adapter: a,
                Domain:  a.Domain(),
            }
        }
    }
    return AdapterResult{Domain: "none"}
}

// Canonicalize calls the resolved adapter.
func (ar *AdapterResult) Canonicalize(tool, op string, rawPayload map[string]any) (*FlatAction, error) {
    if ar.Adapter == nil {
        ar.CanonicalizeMode = "passthrough"
        return nil, nil
    }
    flat, err := ar.Adapter.Canonicalize(tool, op, rawPayload)
    if err != nil {
        return nil, err
    }
    if flat == nil {
        ar.CanonicalizeMode = "rego"
    } else {
        ar.CanonicalizeMode = "go"
    }
    return flat, nil
}

// IntentKey calls the resolved adapter with NORMALIZED payload.
// No CanHandle re-check — adapter was already resolved from raw input.
func (ar *AdapterResult) IntentKey(tool, op string, normalizedPayload map[string]any) (string, error) {
    if ar.Adapter == nil {
        ar.IntentEligible = false
        return "", nil
    }
    key, err := ar.Adapter.IntentKey(tool, op, normalizedPayload)
    if err != nil {
        return "", err
    }
    ar.IntentEligible = key != ""
    return key, nil
}
```

**Why Resolve + reuse, not CanHandle twice:** `CanHandle` is designed
for raw payload shape detection (e.g., "has containers?" → workload).
After canonicalization the shape changes — CanHandle on flat payload
would give wrong results. The registry resolves the adapter ONCE from
raw input, then reuses the same adapter for both canonicalize and
intent key extraction.
```

**Registry order:** K8s workload adapter first (most common case).
Helm, ArgoCD, and Terraform match by tool name — no ambiguity.
Future non-workload K8s adapters (Service, Ingress) would be checked
before K8sWorkloadAdapter to avoid false workload detection.

## 6. Integration with buildActionList

The existing `buildActionList` in `pkg/policy/policy.go` becomes the
integration point:

**Note on `Actor.Origin`:** `buildActionList(params, origin)` uses
origin for exactly one thing: enforcing that MCP origin must use
`params.action` (single action), not `params.actions` (batch).
This is an MCP protocol constraint, not a policy decision. Adapters
never see `actor.*` fields — they operate on payload only.

```go
func (e *Engine) Evaluate(inv invocation.ToolInvocation) (Decision, error) {
    // ... existing input construction ...

    actions, err := buildActionList(inv.Params, inv.Actor.Origin)
    if err != nil {
        return Decision{...}, err
    }

    // NEW: resolve + canonicalize via Go adapters
    var adapterResults []adapter.AdapterResult
    if e.adapterRegistry != nil {
        for i, action := range actions {
            kind := extractString(action, "kind") // e.g. "kubectl.apply"
            parts := strings.SplitN(kind, ".", 2)
            tool, op := parts[0], parts[1]
            payload := extractMap(action, "payload")

            // Resolve adapter from RAW payload (once)
            ar := e.adapterRegistry.Resolve(tool, op, payload)

            // Canonicalize (Go or nil → Rego)
            flat, err := ar.Canonicalize(tool, op, payload)
            if err != nil {
                return Decision{...}, fmt.Errorf("adapter %s: %w", ar.Domain, err)
            }
            if flat != nil {
                actions[i] = flat.ToMap()
            }

            adapterResults = append(adapterResults, ar)
        }
    }

    input["actions"] = actions
    // ... existing OPA evaluation ...

    // AFTER OPA eval: extract intent key from NORMALIZED actions.
    // Uses the same AdapterResult — no second CanHandle dispatch.
    // normalizedActions come from OPA output (defaults.actions).
    for i, ar := range adapterResults {
        normalizedPayload := getNormalizedPayload(opaOutput, i)
        key, _ := ar.IntentKey(tool, op, normalizedPayload)
        // ... feed to deny-loop cache ...
    }
}
```

**When Go adapter returns nil:** Rego canonicalizer runs as before
(K8s workloads). **When Go adapter returns a FlatAction:** Rego
canonicalizer sees already-flat input and passes through (idempotent).

### Phase 0 (v0.3.0): Reorganize + coexist
- Go adapter registry exists
- K8s workloads: intent extraction in Go, canonicalization in Rego
- Helm/ArgoCD/Terraform: full Go adapters (canonicalize + intent)
- Rego `canonicalize.rego` split into `domains/k8s/canonicalize.rego`
  with dispatcher at top level
- All tests green, zero behavior change for existing inputs

### Phase 1 (v0.3.x): Add non-workload K8s adapters as needed
- When Service/Ingress/RBAC policies are added
- These get Go canonicalization (different structures, no podSpec)
- Rego canonicalizer untouched (still handles workloads)

### Phase 2 (v0.4.0+, optional): Full Go canonicalization for workloads
- Only if Rego canonicalizer becomes a maintenance burden
- Move workload podSpec extraction to Go
- Rego canonicalizer becomes passthrough (`actions := input.actions`)
- Currently no reason to do this — Rego works fine for workloads

## 10. Testing Strategy

### Per-domain canonicalization
- **Rego tests** (existing pattern): corpus-driven, assert flat output
- **Go tests** (new): per-adapter, assert FlatAction output matches
  expected flat format for each input shape

### Cross-domain consistency
- Terraform deep fields (from adapter v2) and same fields from MCP
  agent must produce identical flat format
- Test: `TestTerraformAdapter_AgentPayload_MatchesExternalAdapter`

### Intent key stability

**Principle: intent key fields must align with policy triggers.**
The intent key must include every field that can change an allow/deny
result for the domain's destructive policies. Otherwise deny-loop
cache blocks legitimate fixes (e.g., agent fixes `privileged: true` →
`false` but stop_after_deny won't re-evaluate because the key is
unchanged).

Per-domain alignment:
- K8s: namespace, resource, images, security posture (privileged,
  host_pid, host_network, host_ipc, capabilities)
- Terraform: resource_types, destroy_count, SG CIDRs, IAM actions
- Helm: release, namespace, chart, version, values_hash
- ArgoCD: app_name, namespace, project, target_revision

Tests:
- Same input → same key (deterministic)
- Meaningful change → different key (security posture, images)
- Formatting change → same key (casing, whitespace)
- Cross-format → same key (native vs flat K8s, via normalized payload)
- **Policy-trigger change → different key** (critical: fixes must re-eval)

### Guard tests
- Rules must not read raw input directly
- Only adapters and canonicalizer may access raw payload structure
- Boundary guard test scans rule files for forbidden patterns

Adapter-terraform v2 deep extraction is independent of engine v3.
It ships in the `evidra-adapters` repo on its own release cycle.

| Resource Type | Extracted Fields | Flat Output |

| `aws_iam_role_policy` / `_attachment` | policy document actions | `iam_policy_statements[]` |
| `aws_iam_role` | assume_role_policy | `trust_policy_statements[]` |
| `aws_s3_bucket_public_access_block` | block flags | `s3_public_access_block{}` |

**Output contract:** same flat fields that OPA rules already evaluate.
Whether fields come from external CLI adapter (CI pipeline) or MCP
agent (AI extracts into payload), OPA rules fire identically.

## 13. Adapter Telemetry

Every validate response should include adapter metadata in the
decision summary (or a dedicated resource field):

```json
{
    "adapter": {
        "domain": "k8s_workload",
        "canonicalize_mode": "rego",
        "intent_eligible": true
    }
}
```

Fields:
- `domain`: which adapter handled the input (or `"none"` if no match)
- `canonicalize_mode`: `"rego"` | `"go"` | `"passthrough"`
- `intent_eligible`: whether deny-loop cache applies (`true`/`false`)

This is critical for debugging "why was my action blocked" and "why
did deny-loop not prevent retry."

## 14. Non-Overridable Policies and Adapters

Non-overridable policies (the hardcoded rules that cannot be
overridden) rely on specific flat fields produced by adapters.
When adding or modifying non-overridable policies, verify the required
fields are produced by the relevant adapter's Canonicalize output.

Current non-overridable rules and their adapter dependencies:
- `k8s.privileged_container` → requires `containers[].security_context.privileged` (K8s workload adapter via Rego)
- `k8s.host_namespace_escape` → requires `host_pid`, `host_ipc`, `host_network` (K8s workload adapter via Rego)

**Rule:** No non-overridable policy may be added unless the required
fields are present in the adapter's flat output AND tested in the
adapter's test suite. This prevents non-overridable policies that
silently never fire because the adapter doesn't extract the needed field.

**CI enforcement (v0.3.x):** Add a guard test that scans non-overridable
rule files, extracts referenced flat field names, and verifies adapter
test suites cover those fields. Not required for v0.3.0 launch, but
high-ROI addition once the adapter test pattern is established.

## 15. Do NOT

- Do not move K8s workload canonicalization to Go in v0.3.0 (Rego works fine)
- Do not change OPA rule format (rules stay flat, adapters produce flat)
- Do not add k8s.io/api dependency ("structured extraction" means
  manual field paths in unstructured Go maps, not K8s typed structs)
- Do not block v0.3.0 on adapter-terraform v2 (parallel track)
- Do not add Crossplane/CFN adapters in v0.3.0 (future, on demand)
- Do not mix Rego and Go canonicalization for the same resource class
  (workloads owned by Rego; non-workloads owned by Go; each class
  has exactly one canonicalization owner)
- Do not over-engineer Helm values hashing before real usage data
- Do not let adapters read actor.* fields (actor semantics = decision layer)
- Do not hash raw payload as fallback intent key (unstable, bypassable)
- Do not add non-overridable policies without verifying adapter flat
  output includes the required fields
