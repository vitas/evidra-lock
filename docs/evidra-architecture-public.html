1 System Overview

Evidra is a policy-as-code guardrail engine for AI agents and CI pipelines. It evaluates whether a tool invocation — Terraform apply, Kubernetes deploy, database migration — should be allowed, denied, or flagged, then produces a cryptographically signed evidence record.

Evidra follows an API-first hybrid architecture: when a server is available, all clients (CLI, MCP, CI) delegate evaluation to the API for consistent policy and centralized evidence. When offline or in air-gapped environments, clients fall back to a local OPA engine with an embedded policy bundle.

2 Hybrid Mode — API-First with Offline Fallback

Evidra is API-first: when a server is reachable, all evaluation happens server-side. This guarantees consistent policy decisions and centralized evidence across all clients. The local OPA engine serves as an offline fallback for air-gapped environments and network failures.

Online Default

EVIDRA_URL + EVIDRA_API_KEY set, API reachable. All evaluation server-side. Server policy is single source of truth. Evidence in PostgreSQL.

Offline Explicit

No EVIDRA_URL or --offline flag. Local OPA with embedded/custom bundle. Evidence in ~/.evidra/evidence. Fully autonomous, air-gapped.

Fallback Configurable

EVIDRA_URL set but API unreachable. Default: fail closed (error, tool blocked). With EVIDRA_FALLBACK=offline: evaluate locally, mark evidence as "source": "local-fallback".

2.2 Mode Resolution

Env / FlagDefaultEffect
EVIDRA_URL(unset)When set, enables online mode. Example: https://api.evidra.rest (REST API base, client appends /v1/validate)
EVIDRA_API_KEY(unset)Bearer token for API auth. Required in online mode
--offlinefalseForce offline mode even when EVIDRA_URL is set
EVIDRA_FALLBACKclosedclosed = fail with error. offline = evaluate locally when API unreachable
--fallback-offlinefalseSame as EVIDRA_FALLBACK=offline (flag override)
--bundle(embedded)Path to OPA bundle for offline/fallback mode. If unset, uses embedded ops-v0.1
Fail closed by default: When the API is configured but unreachable, Evidra returns an error rather than silently falling back to local evaluation. This prevents policy drift: if you expect server-managed policy, a silent fallback to a stale local bundle could allow actions the server would deny. Opt into fallback explicitly with EVIDRA_FALLBACK=offline.

2.3 MCP Server — Sidecar Pattern

In online mode, evidra-mcp does not make HTTP calls itself. Instead, evidra-api runs as a sidecar on localhost, and the MCP server proxies evaluation through it. This keeps the MCP binary focused on transport (stdio ↔ MCP protocol) while the API handles all business logic.

Why sidecar, not direct HTTP from MCP? Separation of concerns. The MCP binary handles only the MCP protocol (stdio, tool dispatch). The API binary handles auth, tenant isolation, skills, evidence signing, and database writes. If we add features to the API (rate limiting, billing, audit), MCP gets them for free without code changes.

3.3 CI Pipeline → Adapter → API

CI Pipeline — Always Online (API-first)
sequenceDiagram
  participant GH as CI Runner
  participant TF as terraform
  participant AD as evidra-adapter-terraform
  participant API as Evidra API
  participant OPA as Policy Engine

  GH->>TF: terraform plan -out=tfplan.bin
  TF-->>GH: tfplan.bin
  GH->>TF: terraform show -json tfplan.bin
  TF-->>GH: plan.json

  GH->>AD: stdin: plan.json
  AD->>AD: Parse, filter, truncate
  AD-->>GH: stdout: {input, metadata}

  GH->>API: POST /v1/validate
Bearer token
  API->>OPA: Evaluate policy
  API-->>GH: {allow, risk_level, reason}

  alt Allowed
    GH->>GH: Continue to apply
  else Denied
    GH->>GH: Fail CI step
  end
CI is always online. Adapters have no network access — they only transform data. The CI runner handles the API call. There is no offline fallback for CI; if the API is unreachable, the step fails.

The policy bundle is embedded at build time (zero-config). Custom bundles load via --bundle. In online mode, the server's bundle is authoritative. In offline mode, the local embedded or custom bundle is used.

6.2 Rule Examples

# Block mass deletions
deny[msg] {
    input.params.payload.destroy_count > 10
    msg := sprintf("Plan deletes %d resources", [input.params.payload.destroy_count])
}

# Block volume deletion (risk shortcut — no iteration)
deny[msg] {
    "hcloud_volume" == input.params.payload.delete_types[_]
    msg := "Deleting volumes is blocked — data loss risk"
}

# Require review for truncated plans
deny[msg] {
    input.params.payload.resource_changes_truncated == true
    msg := "Truncated plan — requires manual review"
}
