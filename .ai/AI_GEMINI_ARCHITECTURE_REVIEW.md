# Architectural Review: Evidra-MCP

**Reviewer:** Gemini CLI (Senior Software & Systems Architect)  
**Date:** 2026-02-23  
**Status:** Initial Comprehensive Review  

## 1. Executive Summary
Evidra-MCP is a sophisticated, well-structured policy enforcement engine designed for the "AI-Infra" era. It successfully bridges the gap between high-level AI intent and low-level infrastructure execution by introducing a deterministic, auditable middle layer. The choice of OPA/Rego for policy and MCP for integration positions it perfectly for modern developer workflows.

## 2. System Decomposition & Layering

### 2.1 Integration Layer (`cmd/evidra-mcp`, `pkg/mcpserver`)
- **Strengths:** Clean implementation of the Model Context Protocol. The use of standard `stdio` transport ensures compatibility with most AI IDEs (Claude Desktop, etc.).
- **Observations:** The server is tightly coupled to a single policy profile at runtime. While appropriate for v0.1, a multi-tenant or multi-profile capability might be needed for larger organizations.

### 2.2 Evaluation Core (`pkg/validate`)
- **Strengths:** Excellent separation of concerns. The "Scenario" abstraction (`pkg/scenario`) allows the system to remain agnostic of the underlying infrastructure tool (Terraform, K8s, etc.) once the input is normalized.
- **Consistency:** Both the MCP server and the offline CLI use the exact same evaluation logic, ensuring "What you test is what you enforce."

### 2.3 Policy Engine (`pkg/runtime`, `policy/`)
- **Architecture:** Uses OPA as an embedded library. This is the gold standard for cloud-native policy.
- **Contract:** The Rego implementation in `policy/profiles/ops-v0.1/policy/decision.rego` is elegant. It uses a `denies` and `warnings` collection pattern that provides rich feedback (reasons, hits, hints) rather than a simple boolean.

### 2.4 Evidence Store (`pkg/evidence`)
- **Implementation:** Uses a hash-linked, append-only file-based store (JSONL). 
- **Critical Review:** 
    - **Immutability:** While "hash-linked" provides integrity checks, it is "soft" immutability as the files are local and could be deleted or modified (though the hash chain would break).
    - **Concurrency:** Uses file-based locking (`withStoreLock`). This is sufficient for local development but will be a bottleneck if the MCP server is hit by multiple concurrent agents or integrated into a high-throughput CI/CD pipeline.

## 3. Technical Debt & Risks

| Risk Area | Severity | Observation |
| :--- | :--- | :--- |
| **Input Surface** | Medium | `pkg/scenario/load.go` relies on heuristic "looks like" checks. As more formats are added, this could lead to false positives or incorrect parser selection. |
| **Concurrency** | Low/Med | Local file locking for evidence will not scale to multi-user environments. |
| **Policy Versioning** | Medium | There is no explicit versioning or "pinning" of policies within the evidence record beyond a `PolicyRef` string. |
| **Error Handling** | Low | The pipeline is generally robust, but some errors in the MCP layer are "swallowed" and returned as `internal_error` summaries to the client. |

## 4. Architectural Recommendations

### 4.1 Short-Term (Stability & Polish)
1. **Enhanced Evidence Integrity:** Consider signing evidence records with a local private key (optional feature) to move from "hash-linked" to "cryptographically verifiable."

### 4.2 Mid-Term (Capability Expansion)
1. **Pluggable Evidence Backends:** Abstract the `Store` interface further to allow for S3, DynamoDB, or Git-based evidence backends for team/enterprise use.
2. **Contextual Data Injection:** Allow the MCP server to pull external state (e.g., current AWS tag standards, active incident flags) and inject them into the OPA `data` object for more dynamic policies.

### 4.3 Long-Term (Strategic)
1. **Policy Packs:** Move toward a "Plugin" architecture for policies, allowing users to import "AWS Well-Architected" or "SOC2" policy packs without modifying the core `evidra` source.

## 5. Conclusion
The architecture is exceptionally clean and follows modern Go best practices. The "Single Decision Contract" is its greatest strength. By maintaining a strict boundary between the *intent* (Scenario) and the *decision* (Evidence), Evidra-MCP provides a reliable safety net for autonomous infrastructure operations.
